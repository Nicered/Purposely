# PDD(Purpose-Driven Development)란 무엇인가?

## 개요

**목적 중심 개발(Purpose-Driven Development, PDD)**은 프로젝트의 핵심 목적을 개발 생애주기의 중심에 두는 소프트웨어 개발 방법론입니다.

## 핵심 철학

### 문제: 왜 프로젝트는 실패하는가?

대부분의 소프트웨어 프로젝트가 실패하는 이유:

1. **목적 망각**: 개발 도중 "왜 이걸 만드는가?"를 잊어버림
2. **범위 확장**: 원래 목적과 무관한 기능들이 계속 추가됨
3. **일관성 상실**: 각 단계의 결정이 서로 충돌함
4. **우선순위 혼란**: 무엇이 중요한지 판단 기준이 없음

### 해결책: Purpose-First Approach

PDD는 다음과 같은 원칙으로 이 문제를 해결합니다:

1. **단일 진실의 원천**: GLOBAL_PURPOSE가 모든 결정의 기준
2. **지속적 정렬**: 모든 문서가 GLOBAL_PURPOSE를 참조하고 확인
3. **의도적 문서화**: 각 단계의 의도와 근거를 명확히 기록
4. **추적 가능성**: 모든 결정이 목적으로 역추적 가능

## PDD의 핵심 구성 요소

### 1. GLOBAL_PURPOSE

프로젝트의 **존재 이유**를 정의하는 문서입니다.

**필수 질문:**
- **Why**: 왜 이 프로젝트가 존재하는가?
- **Problem**: 어떤 문제를 해결하는가?
- **Solution**: 어떻게 그 문제를 해결하는가?
- **Success Metrics**: 성공을 어떻게 측정하는가?

**예시:**
```markdown
## Why
개발자들이 프로젝트 도중 목적을 잃고 방향성을 상실하는 문제를 해결하기 위해

## Problem
- 프로젝트 시작 시 명확했던 목적이 개발 진행 중 희미해짐
- 기능 추가 시 원래 목적과의 정렬 여부를 확인하기 어려움
- 팀원 간 프로젝트 목적에 대한 이해가 달라짐

## Solution
모든 개발 문서가 GLOBAL_PURPOSE를 참조하도록 강제하는 프레임워크

## Success Metrics
- 프로젝트 완성률 80% 이상
- 목적 이탈 기능 추가율 10% 이하
```

### 2. Phase-Based Development

프로젝트를 **Phase**로 나누어 관리합니다.

각 Phase는:
- 명확한 **목표**(SPEC)를 가짐
- GLOBAL_PURPOSE와 **정렬**되어 있음
- 독립적으로 **완성** 가능함

### 3. 7가지 문서 타입

각 Phase는 다음 순서로 문서를 작성합니다:

```
Phase-01/
├── 00_SPEC.md              # 이 Phase의 목표
├── 01_XX_RESEARCH_*.md     # 조사 및 리서치
├── 02_00_DESIGN_OVERVIEW.md # 전체 디자인 개요
├── 02_XX_DESIGN_*.md       # 세부 디자인
├── 03_PLAN.md              # 구현 계획
└── 04_IMPLEMENTATION.md    # 실제 구현 로그
```

각 문서는:
- **GLOBAL_PURPOSE를 명시적으로 참조**
- 이전 문서들을 기반으로 작성
- 다음 문서의 근거가 됨

## PDD의 장점

### 1. 명확한 방향성
- 모든 팀원이 "왜?"를 이해함
- 의사결정이 빠르고 일관됨
- 우선순위 판단이 명확함

### 2. 범위 관리
- GLOBAL_PURPOSE에 부합하지 않는 기능 거부 가능
- Feature creep 방지
- 리소스 집중

### 3. 팀 협업
- 문서가 공통 언어 역할
- 비동기 협업 가능
- 새 팀원 온보딩 용이

### 4. 품질 보증
- 각 단계의 근거가 문서화됨
- AI가 자동으로 정렬 여부 확인
- 코드 리뷰 품질 향상

## PDD vs 전통적 방법론

| 측면 | 전통적 방법론 | PDD |
|------|--------------|-----|
| 목적 정의 | 초기에만 | 지속적 참조 |
| 문서화 | 선택적 | 필수적 |
| 정렬 확인 | 수동 | AI 자동화 |
| 범위 관리 | 어려움 | 명확함 |
| 팀 이해도 | 불균일 | 일관됨 |

## PDD가 적합한 프로젝트

✅ **적합:**
- 명확한 문제를 해결하는 프로젝트
- 팀 협업이 필요한 프로젝트
- 장기적으로 유지보수될 프로젝트
- 목적이 명확하지만 구현 방법이 불확실한 프로젝트

❌ **부적합:**
- 단순 실험/프로토타입
- 1-2일 완성되는 초소형 프로젝트
- 목적이 계속 바뀌는 탐색적 프로젝트

## 시작하기

PDD를 시작하려면:

1. **GLOBAL_PURPOSE 정의**: "왜?"를 명확히
2. **첫 Phase 계획**: 무엇을 먼저 만들 것인가?
3. **문서 작성**: SPEC → RESEARCH → DESIGN → PLAN → IMPLEMENTATION
4. **지속적 확인**: 각 결정이 GLOBAL_PURPOSE와 정렬되는지 확인

자세한 내용은 [시작하기](getting-started.md) 문서를 참고하세요.

## 결론

PDD는 **"왜?"를 잊지 않는 개발 방법론**입니다.

GLOBAL_PURPOSE를 중심에 두고, 모든 결정이 그것과 정렬되도록 보장함으로써, 프로젝트가 **처음 의도한 목적**을 달성하도록 돕습니다.

> "프로젝트를 완성하는 것보다 중요한 것은, **올바른 프로젝트**를 완성하는 것이다."
