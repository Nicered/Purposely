    모범 사례

    효과적인 Purpose-Driven Development를 위한 실전 팁과 노하우입니다.

    GLOBAL_PURPOSE 작성

    ✅ 좋은 GLOBAL_PURPOSE

    ## Why
개발자들이 프로젝트 중간에 목적을 잊고 불필요한 기능을 추가하다가
프로젝트를 완성하지 못하는 문제를 해결하기 위해

## Problem
- 초기 열정으로 시작한 프로젝트의 70%가 미완성으로 끝남
- 개발 도중 "왜 만드는지"를 잊고 Feature creep 발생
- 팀원마다 프로젝트 목적에 대한 이해가 달라짐

## Solution
모든 개발 문서가 GLOBAL_PURPOSE를 참조하도록 강제하는
문서 중심 프레임워크와 AI 어시스턴트

## Success Metrics
- 사용자 프로젝트 완성률 80% 이상
- 목적 이탈 기능 추가 10% 이하
- 문서 작성 시간 50% 단축

    왜 좋은가: - Why가 명확하고 공감 가능 - Problem이 구체적이고 측정 가능 - Solution이 간결하고 이해하기 쉬움 - Success Metrics가 정량적

    ❌ 나쁜 GLOBAL_PURPOSE

    ## Why
더 나은 개발 도구를 만들기 위해

## Problem
개발이 어려움

## Solution
좋은 도구를 만들 것

## Success Metrics
사용자가 만족함

    왜 나쁜가: - Why가 너무 추상적 - Problem이 구체적이지 않음 - Solution이 모호함 - Success Metrics를 측정할 수 없음

    핵심 원칙

    1. 구체적으로: "더 나은"이 아니라 "무엇이 어떻게 더 나은지"
    2. 측정 가능하게: "만족함"이 아니라 "NPS 50 이상"
    3. 공감 가능하게: 독자가 "맞아, 나도 그래!" 할 수 있게
    4. 한 문장으로 요약 가능하게: 엘리베이터 피치처럼

    Phase 나누기

    ✅ 좋은 Phase 분할

    Phase 1: CLI 기본 인프라 (2주)
- purposely init 명령어
- 템플릿 시스템
- i18n 지원

Phase 2: 문서 생성 명령어 (2주)
- create 서브커맨드
- 7가지 문서 타입 지원
- 테스트 작성

Phase 3: AI 통합 (3주)
- Claude Code 슬래시 커맨드
- 자동 정렬 검증
- 대화형 문서 작성

    왜 좋은가: - 각 Phase가 독립적으로 완성 가능 - 1-3주 단위로 완료 가능 - 명확한 범위

    ❌ 나쁜 Phase 분할

    Phase 1: 모든 기능 구현 (6개월)
- CLI, 웹 UI, AI, 배포, 문서 등등

    왜 나쁜가: - 너무 큼 - 중간 성취감 없음 - 목적 잃기 쉬움

    핵심 원칙

    1. 1-4주 완성: 그 이상은 쪼개기
    2. 독립 완성 가능: 이 Phase만으로 가치 제공
    3. 명확한 경계: In Scope / Out of Scope 명확
    4. 점진적 가치: 각 Phase마다 사용 가능한 결과물

    문서 작성 타이밍

    ✅ 올바른 순서

    1. SPEC 작성
2. SPEC 리뷰 및 승인
3. RESEARCH (필요시)
4. DESIGN 작성
5. DESIGN 리뷰
6. 코드 작성 시작
7. IMPLEMENTATION 동시 작성

    ❌ 잘못된 순서

    1. 일단 코딩
2. 나중에 문서 작성 (절대 안 함)

    핵심 원칙

    1. 문서 먼저: 코드 작성 전 디자인 문서화
    2. 지금 작성: "나중에"는 없음
    3. 동시 업데이트: 코드와 문서는 함께
    4. 리뷰 활용: 문서 리뷰로 설계 검증

    GLOBAL_PURPOSE 정렬 확인

    매 작업 전 질문하기

    새 기능 추가 전:

    1. 이 기능이 GLOBAL_PURPOSE의 어떤 부분을 지원하나?

    2. 대답 못하면 → 필요 없는 기능

    3. 이 기능이 없으면 Success Metrics 달성이 불가능한가?

    4. No → 우선순위 낮음

    5. 이 기능이 Problem 해결에 직접 기여하나?

    6. No → Scope creep

    실전 예시

    상황: "다크 모드 추가하면 좋겠다"

    질문: 1. GLOBAL_PURPOSE의 어떤 부분 지원? - "프로젝트 완성률 향상"과 무관 2. Success Metrics 달성에 필수? - No 3. Problem 해결에 기여? - No

    결론: Phase 1에는 불필요. Nice-to-have는 나중에.

    문서 품질 유지

    SPEC 체크리스트

    SPEC 작성 후 확인:

    • [ ] Objective가 한 문장으로 요약 가능한가?
    • [ ] Scope에 구체적인 산출물이 나열되어 있나?
    • [ ] Out of Scope가 명시되어 있나?
    • [ ] Success Criteria가 체크리스트 형태인가?
    • [ ] GLOBAL_PURPOSE와의 연결이 명확한가?

    DESIGN 체크리스트

    • [ ] 다이어그램이 포함되어 있나?
    • [ ] 공개 API가 명시되어 있나?
    • [ ] 의존성이 명확한가?
    • [ ] 6개월 후 이 문서만 보고 이해 가능한가?

    IMPLEMENTATION 체크리스트

    • [ ] "What Changed"가 솔직하게 작성되었나?
    • [ ] 실패한 시도도 기록했나?
    • [ ] 다음 Phase를 위한 노트가 있나?
    • [ ] 배운 점이 구체적인가?

    Claude Code 활용 팁

    슬래시 커맨드 사용 시점

    상황 사용 커맨드 이유
    프로젝트 시작 /purposely-init GLOBAL_PURPOSE 작성 도움
    Phase 시작 /purposely-phase SPEC 작성 및 검증
    기술 선택 고민 /purposely-research 조사 과정 구조화
    아키텍처 설계 /purposely-design 설계 문서화
    구현 시작 전 /purposely-plan 작업 계획 수립
    구현 중/후 /purposely-implement 로그 작성

    Claude와 대화 팁

    1. 구체적으로 질문: "어떻게 할까요?" 보다 "A vs B 중 뭐가 나을까요?"
    2. 컨텍스트 제공: GLOBAL_PURPOSE를 먼저 보여주기
    3. 반복 개선: 첫 결과가 마음에 안 들면 "더 구체적으로" 요청
    4. 검증 요청: "이게 GLOBAL_PURPOSE와 정렬되는지 확인해줘"

    팀 협업

    문서 리뷰 프로세스

    1. SPEC 리뷰 (Phase 시작 전)
    2. 모든 팀원이 Objective 이해했는지 확인
    3. Out of Scope 합의

    4. Success Criteria 검증 가능한지 확인

    5. DESIGN 리뷰 (코드 작성 전)

    6. 아키텍처 합의
    7. API 설계 검토

    8. 대안 검토

    9. IMPLEMENTATION 리뷰 (Phase 완료 후)

    10. 무엇을 배웠는지 공유
    11. 다음 Phase 개선 사항

    비동기 협업

    문서가 있으면: - 타임존 상관없이 협업 가능 - 새 팀원 온보딩 쉬움 - 의사결정 근거 추적 가능

    팀 컨벤션

    팀에서 합의할 사항: - [ ] 어떤 문서를 필수로 작성할 것인가 - [ ] 문서 리뷰는 누가 언제 하는가 - [ ] 문서 업데이트 책임은 누구에게 - [ ] GLOBAL_PURPOSE 변경은 어떤 프로세스로

    일반적인 실수와 해결책

    실수 1: "나중에 문서 쓰지 뭐"

    문제: - 나중은 절대 안 옴 - 컨텍스트를 잊어버림 - 결국 문서 없이 진행

    해결: - 코드 작성 전 문서 작성을 강제 - PR에 문서 포함 필수화 - "문서 없으면 리뷰 안 함" 규칙

    실수 2: "GLOBAL_PURPOSE 너무 자주 바꿈"

    문제: - 목적이 흔들리면 프로젝트 흔들림 - 팀원 혼란

    해결: - GLOBAL_PURPOSE 변경은 팀 전체 합의 필요 - 정말 근본적 문제만 변경 - 변경 시 모든 문서 영향 분석

    실수 3: "Phase 범위 계속 확장"

    문제: - "하나만 더"의 반복 - 완성하지 못함

    해결: - SPEC의 Out of Scope 엄격히 지키기 - "Phase 2에서" 원칙 - Success Criteria 달성에만 집중

    실수 4: "문서가 코드와 동기화 안 됨"

    문제: - 문서 신뢰 상실 - 결국 안 읽게 됨

    해결: - 코드 변경 시 문서도 함께 업데이트 - PR에 문서 업데이트 포함 - 정기적 문서 감사

    측정과 개선

    추적할 지표

    프로젝트 수준: - Phase 완료율 - 예상 vs 실제 소요 시간 - GLOBAL_PURPOSE 변경 횟수

    개인 수준: - 문서 작성 시간 - 문서 작성 후 코드 재작업 비율 - 목적 이탈 기능 제안 횟수

    회고 질문

    Phase 완료 후: 1. GLOBAL_PURPOSE를 얼마나 자주 참고했나? 2. 작성한 문서 중 가장 유용했던 것은? 3. 문서를 작성하지 않았다면 어떤 문제가 생겼을까? 4. 다음 Phase에서 개선할 점은?

    체크리스트

    프로젝트 시작 시

    • [ ] GLOBAL_PURPOSE 작성 완료
    • [ ] 팀원 전체가 GLOBAL_PURPOSE 이해
    • [ ] Phase 1 SPEC 작성 및 합의
    • [ ] 문서 작성 프로세스 합의

    매 Phase 시작 시

    • [ ] SPEC 작성 및 리뷰
    • [ ] GLOBAL_PURPOSE 정렬 확인
    • [ ] Out of Scope 명확히
    • [ ] Success Criteria 측정 가능한지 확인

    매주

    • [ ] GLOBAL_PURPOSE 다시 읽기
    • [ ] 진행 중인 작업이 목적과 정렬되는지 확인
    • [ ] 문서 업데이트 필요한 것 확인

    Phase 완료 시

    • [ ] IMPLEMENTATION 문서 작성
    • [ ] Success Criteria 모두 달성했는지 확인
    • [ ] 회고 진행
    • [ ] 다음 Phase 계획

    마무리 조언

    "완벽보다 완성"

    문서도 코드처럼: - 처음부터 완벽할 수 없음 - 점진적으로 개선 - 일단 작성하고 개선

    "목적을 잊지 말 것"

    PDD의 핵심: - GLOBAL_PURPOSE가 모든 것의 중심 - 자주 읽고, 자주 확인 - 모든 결정의 기준

    "문서는 팀의 기억"

    6개월 후: - 코드만 보면 이해 못 함 - 문서가 있으면 컨텍스트 복원 가능 - 미래의 팀원을 위한 배려

    축하합니다! 이제 Purpose-Driven Development를 시작할 준비가 되었습니다.
